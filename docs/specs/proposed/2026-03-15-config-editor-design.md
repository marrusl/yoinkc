# Config Editor Design

**Date:** 2026-03-15
**Status:** Proposed
**Scope:** Transform the read-only file browser into an interactive config editor in yoinkc-refine mode.

## Overview

The yoinkc HTML report includes a file browser tab that displays config files, quadlet units, and systemd drop-ins. Today it is read-only and largely redundant with the config/quadlet/drop-in tabs that show the same content inline. This design transforms the file browser into an interactive editor that lets operators modify file content, create new files, and re-render the report—producing export-ready output with all edits baked into the tarball.

The editor is a refine-mode feature only. The static HTML report is unaffected.

## Goals

- Let operators edit config files, quadlet units, and systemd drop-ins directly in the browser.
- Let operators create new files of any of these three types.
- Edits are authoritative: they flow through re-render into the Containerfile and tarball output.
- Eliminate content redundancy between the file browser and other tabs.
- Preserve the static report as a self-contained, dependency-free HTML file.

## Non-Goals

- Editing the generated Containerfile (it is a rendered artifact, not a source).
- Line-level diffs of modifications in the audit report.
- Multi-user collaboration or server-side persistence of edits.

## Architecture

### Client-Side Snapshot Editing

All editing happens in the browser. The report page holds the full inspection snapshot in a `snapshot` JavaScript object (already embedded for the reset-to-original-inspection feature). Edits update this object directly.

- **Save** writes the CodeMirror buffer into `snapshot`—instant, no network round-trip.
- **Re-render** POSTs `{"snapshot": {...}, "original": {...}}` to the `POST /api/re-render` endpoint on yoinkc-refine. This is a breaking change to the existing API contract (currently accepts a bare snapshot JSON body). The server runs yoinkc with `--from-snapshot`, passing both snapshots so the original can be re-embedded and the audit diff can be computed.

This approach requires no new API endpoints for editing. The single source of truth is the in-memory `snapshot` object.

### Original Snapshot Storage

The report page embeds two copies of the snapshot: `snapshot` (mutable, receives edits) and `originalSnapshot` (immutable, never modified). The existing reset-to-original-inspection feature already creates `originalSnapshot` as a deep copy of `snapshot` at page load. This design promotes it to a first-class concept. `originalSnapshot` serves three purposes:

1. **Revert to snapshot** — restores a file's content to its original inspection value.
2. **State detection** — determines whether a file has been modified (content differs from original) or added (not present in original).
3. **Audit diff** — sent alongside `snapshot` in the re-render request so the server can compute the modifications section.

### Preserving originalSnapshot Across Re-renders

The current re-render flow does a full page replacement (`document.open(); document.write(html); document.close()`). The new HTML is generated by yoinkc, which embeds `snapshot` from the template variable `snapshot_json`. Without intervention, `originalSnapshot` would be lost — it would become a deep copy of the modified snapshot.

To preserve it: the re-render request sends both `snapshot` and `originalSnapshot` as a JSON wrapper object `{"snapshot": {...}, "original": {...}}`. yoinkc-refine passes both to yoinkc. The template embeds them as two separate variables:

```javascript
var snapshot = {{ snapshot_json|safe }};
var originalSnapshot = {{ original_snapshot_json|safe }};
```

This replaces the current deep-copy initialization. The original inspection data survives across any number of re-render cycles.

### yoinkc-refine Changes

- **Static asset serving:** Serve the CodeMirror 6 bundle from a `/static/codemirror/` path. This keeps the static report dependency-free while giving refine mode a proper code editor.
- **Refine mode flag:** Inject a `refineMode: true` template variable when serving the report. Templates use this to conditionally render editor UI. Follows the same pattern as the existing fleet mode conditional.
- **Re-render button:** Add a "Re-render" button to the existing yoinkc-refine toolbar (alongside Reset to Original Inspection). Display a count of files in `not rendered` state, e.g., "Re-render (2 files changed)." Only saved files count — `unsaved` files are not included since they haven't been written to `snapshot` yet.

### Data Flow

```
Operator edits file in CodeMirror
  → clicks Save → snapshot.config.files[i].content updated (instant)
  → clicks Re-render → POST /api/re-render with full snapshot
  → yoinkc runs --from-snapshot
  → writes config/, quadlet/, drop-in files with modified content
  → generates Containerfile with COPY directives for all files
  → returns new report.html
  → browser replaces page content
```

Tarball output includes the modified files automatically since yoinkc writes snapshot content to disk.

**Re-render failure:** If the re-render fails (server error, container failure), the browser shows an error banner with the server's error message. All `not rendered` labels remain — no state is cleared on failure. The operator can fix the issue and retry.

**After successful re-render:** All `not rendered` labels clear (the new report reflects the saved snapshot). Any files still in `unsaved` state remain `unsaved` — only saved content was included in the re-render.

## Editor Tab UI

### Two-Pane Layout

The existing file browser layout is preserved: left tree pane, right content pane.

**Tree pane:**
- Displays config files, quadlet units, and systemd drop-ins in a collapsible tree. The current `_build_output_tree()` walks `config/` and `quadlet/` directories; this adds `drop-ins/` as a third root folder.
- A **+ New File** button in the tree toolbar opens the new file creation modal.
- PF6 compact labels on tree entries indicate file state.
- No zebra striping—follows PF6 tree view convention (hover highlights, selected state).

**Content pane — read-only mode (default):**
- Clicking a file in the tree opens it read-only as plain text.
- Toolbar shows the file path, a `read-only` badge, and an **Edit** button.

**Content pane — edit mode:**
- Clicking Edit loads CodeMirror 6 with syntax highlighting and line numbers.
- Toolbar shows the file path, a state label, and three buttons: **Revert to snapshot**, **Save All**, **Save**.
- Clicking another file while the current file has unsaved edits shows a PF6 modal prompting to save, discard, or cancel navigation.

### File State Labels

PF6 compact label pills in the tree and editor toolbar:

| State | Label | Color | Meaning |
|-------|-------|-------|---------|
| Clean | (none) | — | Content matches original inspection, or saved + re-rendered |
| Not saved | `unsaved` | Red (PF6 danger) | Edits in CodeMirror buffer, not yet saved to snapshot |
| Not rendered | `not rendered` | Blue (PF6 info) | Saved to snapshot, report not yet re-rendered |

**State transitions:**

```
Clean → (operator types in CodeMirror) → Unsaved
Unsaved → (clicks Save / Save All) → Not Rendered
Unsaved → (clicks Revert to snapshot) → Clean
Not Rendered → (successful Re-render) → Clean
Not Rendered → (operator types again) → Unsaved
Not Rendered → (clicks Revert to snapshot) → Not Rendered (content reverted to original, but report still shows old rendered content)
Not Rendered → (failed Re-render) → Not Rendered (unchanged)
```

### Toolbar Actions

- **Save** — writes current file's CodeMirror buffer into `snapshot` (instant, no network).
- **Save All** — saves all files with unsaved edits into `snapshot` at once.
- **Revert to snapshot** — restores the current file to its original inspection content from the embedded original snapshot.
- **Edit** — switches from read-only mode to CodeMirror edit mode (read-only mode only).

## Other Tabs — "Edit in Editor" Linking

In refine mode, the config, quadlet, and drop-in tabs replace their inline content pulldowns with a **"View & edit in editor →"** link. Clicking the link navigates to the Editor tab with that file selected in read-only mode. The operator clicks Edit if they want to modify it.

This eliminates the content redundancy (large expandable content blocks that display poorly for big files) while keeping each tab focused on its triage-oriented UI: include/exclude toggles, fleet prevalence, version changes, file kind labels, package badges.

State labels (`unsaved`, `not rendered`) also appear on the card headers in these tabs so the operator knows which files have been modified without switching to the editor.

In the static report, the content pulldowns remain unchanged.

## New File Creation

A PF6 modal dialog triggered by the **+ New File** button. The form adapts based on selected file type.

### Config File

- **File type:** radio group — Config file / Quadlet unit / Systemd drop-in.
- **File path:** text input for the absolute path on the target system (e.g., `/etc/myapp/custom.conf`).
- Creates a `ConfigFileEntry` with `kind: UNOWNED`, empty content, `include: True`.

### Quadlet Unit

- **Scope:** dropdown — "System (root)" or "User — \<username\>" populated from `UserGroupSection.users` in the snapshot.
- **Unit name:** text input (e.g., `myapp.container`).
- **Full path (read-only preview):** assembled from scope and name.
  - System: `/etc/containers/systemd/<name>`
  - User: `/home/<user>/.config/containers/systemd/<name>`
- Creates a `QuadletUnit` with `name` derived from filename, empty `content` and `image`, `include: True`.

### Systemd Drop-in

- **Service:** dropdown populated from discovered services in the snapshot.
- **Override filename:** text input (default: `override.conf`).
- **Full path (read-only preview):** assembled from service and filename.
  - `/etc/systemd/system/<service>.d/<filename>`
- Creates a `SystemdDropIn` with `unit` from selected service, empty `content`, `include: True`.

All types: after clicking **Create & edit**, the file opens immediately in the editor in edit mode with `unsaved` state.

### Validation

- **Path required:** the Create button is disabled until a non-empty path is provided.
- **Absolute path:** config file paths must start with `/`. The form shows an inline error if the path is relative.
- **Duplicate detection:** if a file with the same path already exists in the snapshot, the form shows an inline error ("File already exists — edit it in the Editor tab instead"). No duplicate entries allowed.
- **Quadlet/drop-in filename:** must end with a valid extension (`.container`, `.volume`, `.network`, `.kube`, `.image` for quadlets; `.conf` for drop-ins). Inline error if not.

### Deleting Operator-Created Files

Operator-created files can be removed from the snapshot entirely via a **Delete** button in the editor toolbar (only shown for files that were created by the operator, not for files from the original inspection). Original inspection files can only be excluded via include/exclude toggles, not deleted — this prevents accidental loss of inspection data.

## Audit Report — Modifications Section

A new "Modifications" section at the end of the audit report (both markdown and HTML tab) listing operator changes at the path level:

- **Edited** — files whose content differs from the original inspection.
- **Added** — files created by the operator that did not exist in the original snapshot.
- **Removed** — already tracked via include/exclude.

The section only appears if there are modifications. No line-level diffs—just a list of paths and what happened (edited/added).

Computed server-side during re-render: the client sends both `snapshot` and `originalSnapshot` (see Original Snapshot Storage above) in the re-render request. yoinkc compares the two to identify edited and added files, then generates the modifications list as part of the audit output. This keeps the audit report consistent between markdown and HTML representations and avoids duplicating diff logic in JavaScript.

## CodeMirror 6 Integration

- **Bundle:** CodeMirror 6 basic setup (~150KB minified). Includes syntax highlighting, line numbers, and basic keybindings.
- **Delivery:** Shipped with yoinkc-refine, served as static assets from `/static/codemirror/`. Not embedded in the HTML template.
- **Loading:** The report template conditionally loads CodeMirror JS/CSS via `<script>`/`<link>` tags when `refineMode` is true. Static reports never reference these assets.
- **Instance management:** One CodeMirror instance, reused across files. When the operator selects a different file, the editor content is swapped and undo history is reset (each file gets a fresh undo stack). When switching from read-only to edit mode, CodeMirror is initialized (or shown) for that file.
- **Keyboard shortcuts:** Ctrl+S / Cmd+S triggers Save within the CodeMirror editor.

## Refine Mode Detection

A `refineMode` template variable injected by yoinkc-refine when serving the report. The Jinja2 template uses `{% if refine_mode %}` blocks to conditionally render:

- The Editor tab (instead of the read-only File Browser tab).
- "View & edit in editor →" links (instead of content pulldowns).
- State labels on card headers.
- The Re-render button in the toolbar.
- CodeMirror script/style tags.

The static report receives `refine_mode = False` (or the variable is absent), rendering the original read-only UI.

## Static Report

The static HTML report is completely unaffected by this feature:

- Keeps the read-only file browser tab.
- Keeps content pulldowns in config/quadlet/drop-in tabs.
- No CodeMirror dependency.
- No editor UI, state labels, or "Edit in editor" links.
- Self-contained single HTML file, no external assets.

## Testing Strategy

### Python Unit Tests

- New file creation logic: appending entries to snapshot lists with correct defaults (`kind: UNOWNED` for config, `name` derivation for quadlets, `unit` population for drop-ins).
- Audit report modifications section: generation logic, correct classification of edited/added/removed files, empty-section suppression.
- Refine mode context variable injection.

### Python Integration Tests

- Re-render with modified snapshot content produces correct output files on disk.
- New files appear in Containerfile `COPY` directives after re-render.
- Tarball output includes modified/new file content.

### JavaScript Tests

- Snapshot mutation functions: save (write buffer to `snapshot`), revert (restore from original), save all.
- State tracking: clean → unsaved → not-rendered transitions, correct label updates in tree and toolbar.
- Unsaved-edits navigation guard: modal appears when switching files with unsaved changes; save/discard/cancel all behave correctly.
- New file form: path assembly logic for each type (config path passthrough, quadlet scope-based path, drop-in service-based path), validation rules (duplicate detection, absolute path, valid extensions).
- Refine mode conditional rendering: editor tab present, links replace pulldowns.

### Template Tests

- Refine mode on: editor tab rendered, "Edit in editor" links present, CodeMirror assets loaded.
- Refine mode off: read-only file browser rendered, content pulldowns present, no CodeMirror references.

## Future Considerations

- **Checkboxes → toggle switches:** Consider converting all include/exclude checkboxes to PF6 toggle switches for consistency.
- **Persistence across page refresh:** If needed, a `POST /api/snapshot` endpoint could persist the in-memory snapshot to disk (Approach 3 from brainstorming). Layerable without rearchitecting.
- **Syntax-aware editing:** CodeMirror language modes for INI, TOML, systemd unit files. Can be added as additional CM6 extensions.
